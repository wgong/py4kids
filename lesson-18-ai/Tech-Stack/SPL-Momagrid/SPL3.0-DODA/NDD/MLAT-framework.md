# Multi-Level Alignment Test (MLAT)
## Measuring Intent Preservation Across the Full Transformation Pipeline

**Author:** Wen G. Gong
**Date:** April 16, 2026
**Status:** Research note — extends NDD closure to the full intent-to-deployment chain
**Companion documents:**
- `NDD-closure.md` — formal definition of NDD closure; SPL implementation; real-world validation
- `NDD-position.md` — philosophical grounding; Noether analogy; agentic integrity lineage

---

## 1. Motivation

The existing NDD closure property answers one question:

> Did the code generation system faithfully encode the original specification?
> `J(S, E(G(S))) = CLOSED?`

This is necessary but not sufficient. A production agentic system undergoes multiple
transformations between the moment a human forms an intent and the moment behavior appears in
the world:

```
Human intent (S)
    │  text2SPL / authoring
    ▼
.spl specification (C)
    │  splc compiler
    ▼
Target language artifact (T)     ← Go / TypeScript / Python
    │  deploy to device
    ▼
Runtime traces and logs
```

Each arrow is a transformation that **claims to preserve the original intent S**.
Each claim is independently testable. Each failure reveals a different class of problem.

**Multi-Level Alignment Test (MLAT)** applies a measurement at every transformation
boundary, producing a multi-dimensional alignment profile for an agentic system.

**Terminology:**
- **MLAT** — the four-level test framework; the measurement instrument
- **NDD Closure** — the achieved state when all four MLAT levels pass; the totality condition

---

## 2. Physical Grounding

The framework is grounded in physical measurement methodology. The four MLAT levels
correspond directly to the four experimental stances a physicist takes when validating
a theory:

| MLAT level | Physics analogue |
|-----------|-----------------|
| J₁ Ideation | Thought experiment — does the theory encode the physical intuition correctly? |
| J₂ Design-time | Blueprint verification — does the formal model match the theoretical intent? |
| J₃ Runtime | Controlled laboratory measurement — deterministic, variables isolated |
| J₄ Behavior | Field observation — deployed in the real world; statistical ensemble |

**S as the conserved invariant.** By Noether's theorem, every continuous symmetry of
a physical system corresponds to a conserved quantity. MLAT applies the same logic: if
the round-trip symmetry `extract(transform(S)) ≈ S` holds at every level, then S — the
original human intent — is conserved across the full transformation chain. If it breaks
at any level, something has leaked and the measurement tells you exactly where.

**J₃ is controlled experiment.** The NDD echo oracle (`spl3 run --adapter echo`)
eliminates the LLM variable entirely, holds all inputs constant, and measures control
flow deterministically. This is experimental isolation: one variable changed, all others
fixed. The measurement is reproducible.

**J₄ is statistical mechanics.** Deployed systems are open, non-deterministic, and
generate large trace volumes. No single observation is definitive. J₄ samples the
ensemble, extracts the behavioral distribution, and measures drift from the original
specification. The appropriate tool is statistical sampling, not exact measurement —
the same shift from laboratory to field that every experimental science makes.

## 3. The Four-Level Pipeline

```
S ──── G ────► C ──── splc ────► T ──── deploy ────► logs/traces
│              │                 │                    │
│ ◄── S₁ ──── │                 │                    │
│              │ ◄── S₂ ──────── │                    │
│              │                 │ ◄── S₃ ────────── │
│              │                 │                    │ ◄── S₄
│              │                 │                    │
J₁(S,S₁)    J₂(S,S₂)         J₃(S,S₃)           J₄(S,S₄)
Ideation     Design-time      Runtime             Behavior
```

Where:
- **S** — original human intent, expressed in natural language
- **C** — the `.spl` specification (authored or generated by `text2SPL`)
- **T** — target-language artifact produced by `splc` (Go, TypeScript, Python/LangGraph)
- **S₁, S₂, S₃, S₄** — intent reconstructions extracted at each level
- **J₁–J₄** — alignment judges, one per transformation boundary

Human judgment is present at every level. The judges are measurement instruments, not
oracles. The human is the governing authority at all levels — the same role a physicist
plays when deciding whether a measurement result is physically meaningful.

## 4. The Four Judges

### J₁ — Ideation Judge

```
S  ──[G]──►  C
             │
             └──[LLM]──► S₁ = LLM(C)

Measurement: J₁(S, S₁)
```

**The transformation:** Human intent `S` → `.spl` artifact `C` via `text2SPL` or manual
authoring.

**The probe:** Ask an LLM to read `C` and express what it understands the workflow to do.
The result is `S₁`.

**What it measures:** Did the authoring step preserve the intent? If `J₁` diverges, the
`.spl` is wrong before any compilation or deployment — the earliest and cheapest divergence
to catch.

**Physics analogue:** Thought experiment consistency check. Does the mathematical
formulation actually encode the physical intuition the author intended?

**Instrument:** LLM semantic comparison. Human review governs.

---

### J₂ — Design-time Judge

```
C  ──[E]──► S₂ = E(C)

Measurement: J₂(S, S₂)
```

**The transformation:** `E` is formal spec extraction — reading `C` and producing a
functional description `S₂` of what the workflow does, independent of how `C` was
produced.

**What it measures:** Is the `.spl` code, as written, a faithful design-level encoding of
the intent? This is the original NDD closure property (`J(S, E(G(S))) = CLOSED`) applied
at the `.spl` layer.

**Physics analogue:** Blueprint verification. Does the formal model (equations, diagrams)
accurately represent the theoretical intent?

**Instrument:** LLM semantic comparison — `extract_spec.spl` + `spec_judge.spl` from
`cookbook/56_code_pipeline/` (already implemented). Verdict sentinels: `[CLOSED]` /
`[DIVERGED]` (inheriting NDD-closure.md vocabulary for this level).

`[DIVERGED]` from J₂ is a **hard stop**. The `.spl` must be corrected, or U must
clarify whether the divergence reflects specification ambiguity (U refines S) or system
error (pipeline is corrected). A system that silently rewrites `S` to match `C` has
destroyed the accountability chain.

---

### J₃ — Runtime Judge

```
C  ──[splc]──► T
               │
               └──[LLM / echo oracle]──► S₃

Measurement: J₃(S, S₃)
```

**The transformation:** `splc` compiles `.spl → T`. The claim is semantic preservation:
`T` implements the same workflow as `C`.

**What it measures:** Did `splc` introduce drift during compilation? A `[DIVERGED]` here
means the compiler failed — not the author and not the deployment environment.

**The deterministic instrument:** Run `C` under `spl3 run --adapter echo` and `T` under
the equivalent echo adapter. Compare control flow, branching, and output routing. This is
a fully deterministic, reproducible measurement — no LLM, no sampling. The echo oracle
eliminates the non-deterministic variable entirely.

**Physics analogue:** Controlled laboratory measurement. All variables isolated except the
one under test (`splc`). Reproducible. Exact.

The echo oracle is J₃'s primary instrument. LLM semantic comparison of `T → S₃` is the
secondary instrument, catching semantic drift that structural comparison misses.

**DODA guarantee:** J₃ is what makes "Design Once, Deploy Anywhere" a falsifiable claim
rather than a design aspiration.

---

### J₄ — Behavior Judge

```
T  ──[deploy]──► runtime
                 │
                 └── traces / logs
                           │
                           └──[LLM]──► S₄

Measurement: J₄(S, S₄)
```

**The transformation:** `T` operates in the real world. Observable evidence accumulates
as traces, logs, and outputs.

**What it measures:** Does the system's *runtime behavior* match the original intent?
`[DIVERGED]` at this level means behavioral drift — the deployed system is doing something
other than what was specified.

**The statistical instrument:** Deployment generates large, heterogeneous data volumes.
No single trace is definitive. J₄ samples a representative window, extracts the
behavioral pattern via LLM analysis, and compares against `S`. Trend analysis — is the
alignment score improving or declining over time? — is as informative as the point estimate.

A sustained decline in J₄(S, S₄) is the early-warning signal of behavioral drift before
it becomes a production incident.

**Physics analogue:** Field observation / statistical mechanics. Open system, uncontrolled
environment, ensemble measurement. The appropriate method is sampling, not exact
measurement. The same shift that separates particle physics in a collider from atmospheric
chemistry in the real world.

**Instrument:** Statistical sampling + LLM behavioral analysis. Instrumentation depth
and sample window are tunable parameters governed by the operational team.

## 5. Diagnostic Power of the Combination

The four judges are not redundant — each isolates a different class of failure:

| Judge | Catches failures in | Controlled by |
|-------|--------------------|--------------------|
| J₁ | Intent authoring / `text2SPL` | Author / `text2SPL` system |
| J₂ | `.spl` specification quality | Author / designer |
| J₃ | `splc` compiler correctness | `splc` transpiler |
| J₄ | Deployed runtime behavior | Execution environment, model, inputs |

**Localization principle:** If J₂ passes but J₃ diverges, the bug is in `splc`.
If J₃ passes but J₄ diverges, the deployment environment is the source.
The multi-level profile turns "something is wrong" into "the problem is in layer X."

**NDD Closure** is the conjunction:

```
NDD Closure(S) ⟺  J₁(S,S₁) = ALIGNED  ∧
                   J₂(S,S₂) = ALIGNED  ∧
                   J₃(S,S₃) = ALIGNED  ∧
                   J₄(S,S₄) ≈ ALIGNED     (statistical approximation)
```

The word "Closure" is reserved for this totality. A system that passes all four MLAT
levels has achieved NDD Closure across its full pipeline.

## 6. Engineering Interpretation

MLAT is a measurement framework. It makes no claims about the nature of the system
being measured — whether it is a software pipeline, a compiler toolchain, or any
other transformation chain that claims to preserve a specification.

The framework asks one question at each boundary:

> Can you recover what went in from what came out?

This is the same question asked of any information-preserving system:
- A lossless codec: `decode(encode(x)) = x`?
- A compiler: does the binary preserve the source semantics?
- A translation: does back-translation recover the original?
- A scientific instrument: does re-measurement recover the original signal?

MLAT applies this canonical engineering test to the intent-to-deployment pipeline,
at four boundaries, with measurement methods appropriate to the nature of each boundary
(deterministic where controllable, statistical where not).

No theory of agent consciousness is required. No philosophical claims about AI are
necessary. The measurement stands on its own as an engineering criterion.

The existing SPL v3.0 implementation covers J₂ and J₃ (echo oracle). J₁ becomes
implementable once `text2SPL` ships. J₄ requires the `splc judge` command and trace
collection infrastructure.

## 9. Summary

MLAT is a four-level measurement framework for intent preservation across the
full intent-to-deployment pipeline:

- **J₁ (Ideation Judge):** Did authoring preserve the intent? *(LLM semantic comparison)*
- **J₂ (Design-time Judge):** Does the `.spl` faithfully encode the intent? *(LLM, hard stop on divergence)*
- **J₃ (Runtime Judge):** Did `splc` preserve semantics? *(deterministic — echo oracle)*
- **J₄ (Behavior Judge):** Does the deployed system behave as intended? *(statistical — trace sampling)*

The measurement methodology mirrors physical experimental practice:
thought experiment → blueprint → controlled lab → field observation.
J₃'s deterministic instrument (echo oracle) is the controlled experiment.
J₄'s statistical instrument is the field observation.

**NDD Closure** is the state achieved when all four MLAT levels pass — the totality
condition that the full pipeline preserves the original human intent from ideation to
deployed behavior.

## 10. NDD as AI-Assisted SDLC — The Iterative Development Cycle

### 10.1 S is not optimal: U as the first mover

Steve Jobs observed that users often cannot articulate what they want — and that what they
say rarely matches what they mean. This is not a flaw in the user; it is the nature of
intent formation. Clarity emerges through iteration, not declaration.

This observation has a direct consequence for MLAT: **S is not the platonic ideal of
intent — it is the user's current-best expression of intent.** There is a first mover
U (the user, the human) who produces S:

```
U ──► S
```

S is lossy at the source. The transformation U → S introduces ambiguity, incompleteness,
and unstated assumptions before any system has touched the pipeline.

### 10.2 J_i divergence as ambiguity signal

When a judge returns `[DIVERGED]`, there are two possible root causes:

1. **System drift** — the transformation introduced semantic error (the LLM, the compiler,
   the deployment environment changed the meaning)
2. **Specification ambiguity** — S was underspecified; the system materialized one valid
   interpretation, and it was not the one U intended

MLAT cannot distinguish these causes from the divergence signal alone — but U can.
Showing the divergence report to U is therefore not just an error notification;
it is a **clarification prompt**:

> "This is what the system understood your specification to mean. Is this what you intended?"

U's reaction reveals the missing precision. The parts of S that were ambiguous are exactly
the parts where the system's interpretation surprises the user. **Ambiguity in S is
information** — and divergence reports surface it.

### 10.3 The full iterative loop

With U as the first mover and divergence reports fed back as clarification prompts, the
pipeline becomes a closed development cycle:

```
U ──► S ──[G]──► C ──[splc]──► T ──[deploy]──► logs
↑     │           │              │               │
│     J₁         J₂             J₃             J₄
│                                                │
└──────────── divergence report ─────────────────┘
         "Is this what you meant?"
         U refines S → next iteration
```

This is an iterative development cycle. S evolves through rounds of implementation and
measurement. NDD Closure is not a one-shot test — it is the **convergence criterion** of
an iterative process, reached when U's intent is fully expressed and the full pipeline
preserves it without drift.

### 10.4 NDD as TDD extended to the requirements layer

Test-Driven Development (TDD) made tests the primary specification artifact — written
first, governing everything downstream. NDD extends this discipline one layer up:

| | TDD | NDD |
|--|-----|-----|
| Primary artifact | Test (formal, executable code) | Spec S (natural language) |
| Who authors it | Developer | User U |
| "Red" | Test fails | J_i diverges |
| "Green" | Test passes | J_i = ALIGNED |
| "Refactor" | Improve code, keep tests green | Refine S, re-run MLAT |
| Iteration driver | Failing test | Divergence report shown to U |
| Assumption about spec | Tests are correct as written | S evolves; U → S is itself iterative |

The critical difference: TDD assumes the test is correct — the developer wrote it with
full knowledge of the requirement. NDD acknowledges that S itself is under refinement,
because U is not omniscient about their own intent. This is not a weakness; it is a more
honest model of how specifications actually evolve in practice.

**NDD is TDD for the requirements layer, with U as the test author and MLAT as the
test harness.**

### 10.5 NDD as AI-assisted SDLC

The iterative NDD cycle maps directly onto the Software Development Life Cycle:

| SDLC phase | NDD equivalent |
|-----------|----------------|
| Requirements elicitation | U → S (first expression of intent) |
| Requirements review | J₁: ideation alignment check |
| Design specification | S → C (authoring / `text2SPL`) |
| Design review | J₂: design-time alignment check |
| Implementation | C → T via `splc` |
| Unit / integration testing | J₃: runtime alignment (echo oracle) |
| Deployment | T → runtime |
| Production monitoring | J₄: behavioral alignment (statistical) |
| Sprint retrospective | Divergence report → U refines S → next iteration |

Every phase of a traditional SDLC has a direct NDD equivalent. NDD is not a replacement
for SDLC — it is **SDLC adapted for AI-assisted development**, where the implementation
engine is a combination of LLMs and deterministic compilers (`splc`), and the test harness
is MLAT rather than a unit test suite.

The same iterative cadence applies: short cycles, early feedback, evolving requirements.
The same convergence criterion applies: the system does what the specification says.
The medium changes — natural language replaces formal requirements documents — but the
discipline is identical.

### 10.6 Positioning for the paper: incremental, not revolutionary

Human cognition resists radical novelty but accepts incremental extension. The history
of successful technical standards confirms this: TCP/IP extended packet switching; SQL
extended relational algebra; TDD extended unit testing with a workflow discipline. None
asked practitioners to abandon what they knew.

The paper introduction of NDD follows the same strategy:

> **NDD extends TDD into the AI-assisted SDLC era.** TDD made tests the first-class
> specification artifact at the implementation layer. NDD extends this discipline one
> layer up — to the requirements layer — using natural language as the specification
> medium and MLAT as the test harness. SPL is the reference test-bed: a declarative
> orchestration language that makes the full NDD cycle implementable in under 40 lines.

This framing:
- Anchors to TDD, which has universal acceptance
- Claims extension, not replacement — zero abandonment cost
- Grounds the claim in a concrete, runnable artifact (SPL cookbook recipes)
- Introduces MLAT as the natural generalization of "run the tests"

The audience who already practices TDD will recognize the discipline immediately.
The audience who manages SDLCs will recognize the lifecycle mapping immediately.
Neither needs to accept a new paradigm — they are being shown the next step in
something they already believe in.
